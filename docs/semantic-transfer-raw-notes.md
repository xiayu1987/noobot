# semantic-transfer 原始说明（对照稿）

> 说明：本文件按原始描述整理，仅做排版，不改动语义，供后续方案对照。

## 0. 统一入口原则（新增）

- semantic-transfer 对外只暴露一个统一方法。
- 本文所有场景（工具信息传递、workflow 子agent 信息流转、harness 消息管理、工作流最终返回补充）都必须通过这个统一方法进入 semantic-transfer。
- 各个场景调用统一方法时，只是传入的场景类型、策略和上下文不同；不再按场景分别暴露多个方法。
- 统一方法只负责语义信息转换、附件落盘/引用生成、注入消息生成和标准语义返回，不把插件/业务流程本身耦合进 semantic-transfer。

统一方法：语义信息传递  
输入：场景类型、策略、待转换信息、附件/注入目标等上下文。  
输出：标准 semantic-transfer 语义返回（如 transferResult / transferEnvelopes），由调用方按场景回填。

## 1. 工具信息传递

### 工具输出信息

- 输入是：传入工具的原始返回。
- 输出是：如果原始返回超过限制，就保存到附件并返回附件地址；如果原始返回没有超过限制，就返回原始返回。
- 现状：应该是实现了，但是细节是否考虑优化。
- 建议：通过统一方法调用，策略为“工具信息传递”。

### 工具输入信息

- 主要针对：文件保存工具、脚本执行工具。
- 如果传入的脚本或文件内容超过限制，就保存到附件，然后直接返回超过限制内容以保存到附件，并且带上附件地址。
- 小结工具补充：`task_summary.summaryContent` 也属于工具输入语义信息传递场景；小结内容需要强制通过 semantic-transfer 落盘为附件，并返回标准 semantic-transfer 语义字段（如 transferResult / transferEnvelopes）。该强制附件场景不是超长溢出，不应返回 callOverflow / toolInputOverflow。
- 现状：应该是实现的，但是好像没在 semantic-transfer 层实现。
- 建议：通过统一方法调用，策略为“工具信息传递”。

## 2. 工作流插件子agent信息流转

- 注意：中间有汇聚/分叉等状态节点，通过工作流类库进行扩展，返回实际的下一步步骤。

情况 1：单到单  
单子agent完成，子agent最后信息，强制保存为附件，并且植入附件信息到下一个直接连接的子agent，附件信息成为一条消息。

情况 2：多到单  
并发多子agent所有子agent最后信息，强制每个信息保存为附件，并且植入附件信息到下一个直接连接的子agent，附件信息成为一条消息。

情况 3：单到多  
单子agent完成，子agent最后信息，强制保存为附件，并且植入附件信息到下面所有直接连接的子agent，附件信息每个子agent一条消息。

情况 4：多到多  
并发多子agent所有子agent最后信息，强制每个信息保存为附件，并且植入附件信息到下面所有直接连接的子agent，附件信息每个子agent一条消息。

- 失败传播补充（新增）：
  - 如果上游子agent失败，除了已有附件信息，还要把“上游任务 + 失败信息”植入到直接下游子agent系统消息中。
  - 适用于四种拓扑：单到单、多到单、单到多、多到多。
  - 若上游失败且无附件，下游仍需基于失败信息继续可完成部分，并明确受影响范围。

- 现状：应该有这种逻辑，但是不知道是不是 semantic-transfer 层在处理。
- 建议：通过统一方法调用，策略为“子agent 信息传递”。
- 输入子agent或多个子agent最后信息，输出附件地址。

## 3. harness插件 harness消息管理

- agent 执行完成后的最终回复不再拼接最后一次完整小结和验收单；小结与验收报告只保留在 harness 内部状态、日志或附件/transfer payload 中，不直接追加到最终消息正文。
- agent 触发小结流程：将小结后信息保存到附件，并按参数决定注入主agent的信息形态：默认注入小结后的全量消息；当参数显式要求摘要模式时，才只注入摘要。
- 现状：最终消息拼接小结/验收单的需求已取消；阶段小结附件落盘与主agent注入仍需要通过 semantic-transfer 统一方法处理。
- 建议：通过统一方法调用；阶段消息处理等场景通过不同策略调用该方法，不再分别暴露多个方法。最终消息正文不再作为 semantic-transfer 拼接目标。

策略 1：最终消息拼接（已取消 / 不再实现）  
输入：最终消息、所有小结信息、验收单。  
输出：不再修改最终消息正文；如需保留明细，走附件/transfer payload 或 harness 内部报告。  
状态：废弃。前端也不渲染历史遗留的 harness 小结/验收折叠块。

策略 2：阶段消息处理  
输入：小结消息（全量消息、摘要、明细、注入模式参数）。  
输出：明细附件地址 + 主agent注入消息；注入模式默认 full，注入小结后的全量消息，只有参数显式为 summary 时才注入摘要。  
（注意和插件解耦，语义化参数及方法，摘要和明细解析在插件端做，保存附件与注入消息生成在 semantic-transfer 层做）

## 4. 补充约束（新增）

- semantic-transfer 对外只暴露一个统一入口方法；本文所有场景都必须通过该统一入口，工具信息传递、workflow 子agent信息流转、harness消息管理、工作流最终返回等只是在入参策略上不同。
- semantic-transfer 最后返回要精简，优先复用现有方法（如 compact 相关方法），避免把大文本/冗余字段再次注入主链路。
- 不要越权：semantic-transfer 只处理本说明中的三类场景（工具信息传递、workflow 子agent信息流转、harness消息管理）。
- 其他不在本说明内的场景，统一视为 out of scope，不在本次改造内扩展；已出现的越权实现要收敛/移除到边界外。
- 最后阶段要去掉兼容输出（legacy 字段），统一只保留 semantic-transfer 标准语义返回。
- semantic-transfer 主要只负责信息转换。
- 我不是要所有附件都走 semantic-transfer，而是我场景需要转换的内容才走 semantic-transfer。

## 5. 工作流规划最终返回补充（新增）

- 工作流插件在“工作流规划 + 执行完成”后的最终返回，会拼接各个子agent附件消息（附件路径/摘要块）。
- 这一步拼接结果必须走 semantic-transfer 统一方法（按授权场景，策略归到“子agent 信息传递”或“workflow 最终返回拼接”），不能绕过 semantic-transfer 直接裸拼接返回。
- 如果当前实现没有走 semantic-transfer，需要补齐：由 semantic-transfer 产出标准 transferResult / transferEnvelopes，再回填到 workflow 最终消息与 payload。
